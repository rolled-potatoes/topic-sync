# Feature Specification: Topic Catalog Synchronization

**Created**: 2026-04-29
**Status**: Draft
**Input**: 토픽을 config에서 각 필드별 관리를 하고 있음. 서비스에서는 토픽을 실제 사용도 해야되니 제너레이터로 하여 제공하며 주입하여 sync를 맞춰도 좋겠음.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Keep Service Topics Consistent (Priority: P1)

서비스 운영자는 설정에서 관리되는 토픽 목록과 서비스에서 실제 사용하는 토픽 목록이 항상 일치하기를 원한다. 이를 통해 설정과 실제 사용 간 불일치로 인한 메시지 처리 오류를 줄일 수 있다.

**Why this priority**: 토픽 불일치는 메시지 송수신 실패, 데이터 누락, 운영 장애로 직결되므로 가장 우선순위가 높다.

**Independent Test**: 설정 토픽 목록을 기준으로 서비스가 사용하는 토픽 목록을 생성/반영한 뒤, 양쪽 목록이 1:1로 일치하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 설정에 유효한 토픽 항목이 정의되어 있고 서비스가 토픽을 참조할 준비가 된 상태에서, **When** 서비스가 토픽 목록을 사용한다, **Then** 서비스는 설정에 존재하는 토픽만 사용한다.
2. **Given** 설정에서 특정 토픽이 추가 또는 변경된 상태에서, **When** 서비스가 최신 토픽 목록을 반영한다, **Then** 서비스의 사용 토픽도 동일하게 갱신된다.

---

### User Story 2 - Detect and Prevent Topic Drift (Priority: P2)

운영/품질 담당자는 서비스가 설정에 없는 토픽을 사용하는 경우를 조기에 식별하고 수정하기를 원한다. 이를 통해 배포 전후의 설정-서비스 불일치 리스크를 줄인다.

**Why this priority**: 운영 안정성과 변경 추적성을 높이지만, 핵심 사용 흐름(정합성 유지) 다음 우선순위다.

**Independent Test**: 설정 목록과 서비스 사용 목록을 비교하여 누락/초과/이름 불일치가 있을 때 검출 결과가 명확히 보고되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 서비스가 설정에 없는 토픽을 참조하는 상태에서, **When** 정합성 검증이 수행된다, **Then** 불일치 항목이 식별 가능하게 표시된다.
2. **Given** 설정과 서비스 토픽이 완전히 일치하는 상태에서, **When** 정합성 검증이 수행된다, **Then** 불일치 없음 결과가 확인된다.

---

### Edge Cases

- 설정에 토픽 항목이 비어 있거나 최소 필수 항목이 누락된 경우 어떻게 처리되는가?
- 동일 의미의 토픽이 중복 정의되어 충돌할 때 시스템은 어떻게 일관된 기준을 적용하는가?
- 설정에는 존재하지만 현재 서비스에서 미사용인 토픽이 있을 때 어떤 상태로 분류되는가?
- 서비스가 과거 토픽 이름을 계속 참조하는 경우 호환성/검증은 어떻게 처리되는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 설정에서 관리되는 토픽 정보를 단일 기준 소스로 간주해야 한다.
- **FR-002**: 시스템은 설정 기준 소스에서 서비스가 사용할 토픽 목록을 일관된 형태로 제공해야 한다.
- **FR-003**: 시스템은 서비스가 토픽을 사용할 때 설정 기준과 동기화된 최신 목록을 참조할 수 있어야 한다.
- **FR-004**: 시스템은 설정에 없는 토픽이 서비스에서 사용될 경우 이를 식별 가능하게 표시해야 한다.
- **FR-005**: 시스템은 설정에 새 토픽이 추가되거나 기존 토픽이 변경될 때 서비스 사용 목록에 반영 가능해야 한다.
- **FR-006**: 시스템은 토픽 동기화 상태(일치/불일치)를 검증 가능한 형태로 제공해야 한다.
- **FR-007**: 시스템은 불일치 검출 시 운영자가 조치 가능한 수준의 식별 정보(대상 항목)를 제공해야 한다.
- **FR-008**: 시스템은 토픽 목록 제공 및 검증 과정에서 기존 정상 토픽 사용 흐름을 중단시키지 않아야 한다.

### Key Entities

- **Topic Definition**: 설정에서 관리되는 토픽 항목. 주요 속성은 식별 이름, 용도 구분, 상태(활성/비활성) 등이다.
- **Service Topic Catalog**: 서비스가 실제 참조하는 토픽 목록. Topic Definition에서 파생되며 정합성 검증 대상이다.
- **Synchronization Status**: 설정 목록과 서비스 목록의 일치 여부를 나타내는 상태 정보. 일치, 누락, 초과, 충돌 등의 결과를 포함한다.

## Assumptions

- 토픽의 권한/보안 정책은 기존 운영 정책을 그대로 따른다.
- 성능 기대치는 일반적인 서비스 설정 갱신 및 검증 주기에서 사용자 체감 지연이 없는 수준을 목표로 한다.
- 불일치 검출 결과는 운영자가 확인 가능한 표준적인 형태로 제공된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 설정 변경 후 다음 검증 주기 내에 서비스 토픽 목록과의 일치율이 100%로 확인된다.
- **SC-002**: 설정에 없는 토픽 사용 사례는 검증 수행 시 100% 식별된다.
- **SC-003**: 운영자가 불일치 항목을 확인하고 조치 대상을 특정하는 데 5분 이내가 소요된다.
- **SC-004**: 릴리스 검증 과정에서 토픽 불일치로 인한 운영 이슈 발생 건수가 기준 기간 대비 최소 50% 감소한다.
